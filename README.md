# 快递助手 ERP → 飞书利润与工资同步

该项目把快递助手 ERP 的订单、售后、平台商品、ERP 货品、库存和店铺毛利同步到现有飞书多维表格，并在每月 15 日按 `13_人员表` 自动结算上月工资。

## 已采用的计算口径

- 提成基数：ERP 店铺毛利报表的最终净利润 `netSalesProfit`。
- 退款与退货：由 ERP 毛利报表统一处理，飞书工资表不重复扣除。
- 亏损店铺：提成固定为 0。
- 工资：`底薪 + 绩效工资 + 奖金 + MAX(0, 店铺利润) × 提成比例 - 扣款`。
- 结算：每月 15 日结算上月，保留两位小数，保存利润/提成快照并锁定结果。
- 参数更新：人员表修改店铺、底薪或提成比例后，系统自动刷新所有“待结算”工资；“已结算”历史不变。
- 重启补偿：服务每次启动会先同步订单/售后/库存，再补跑商品、利润和工资任务，避免错过凌晨任务。

客户只需维护人员表里的店铺、底薪、提成比例，以及工资表每月的绩效工资、奖金、扣款。

## 安装与配置

要求 Node.js 18.17 或更高版本，无第三方运行依赖。

1. 复制 `.env.example` 为 `.env`。
2. 填写飞书应用的 `FEISHU_APP_ID`、`FEISHU_APP_SECRET`。
3. 填写快递助手的 `KDZS_APP_KEY`、`KDZS_APP_SECRET`；已有 session 时也可以只填 `KDZS_SESSION`。
4. 根据实际经营开始日期调整 `SYNC_START_DATE`。

不要把 `.env` 提交到版本库或发到群聊；该文件已被 `.gitignore` 排除。

## 运行

```powershell
npm test
npm run check
npm run migrate
npm run sync:full
npm run sync
npm run payroll
# 历史全量回填（建议通过 GitHub Actions 手工触发）
node src/index.js new-backfill --start=2026-06-01
node src/index.js new-archive
```

- `check`：实际查询一条库存，同时验证快递助手 session、网关、签名和飞书读取权限。
- `migrate`：只补充同步键、同步时间和工资锁定字段，不拉取 ERP 数据。
- `sync:full`：首次全量导入。
- `sync`：日常增量同步，并在符合日期时自动结算工资。
- `payroll`：单独执行工资月结；测试时可用 `node src/index.js payroll --force`。
- `new-backfill`：按 ERP 接口可查询范围回填订单、售后、物流、订单/商品利润；生产环境建议使用 GitHub Actions 的 `Full ERP backfill`，避免占用 Render Web 进程。
- 原始订单、订单明细和物流按创建月份直接写入分表；单月接近15,000条时自动切换为该月上、下半月分表，避免飞书单表20,000条上限。

每次同步会在 `logs` 目录生成 JSON 报告，包含各表读取、新增、更新、失败数量和失败原因。

## 增量与防重策略

- 订单：`系统订单号 + 子订单号`。
- 售后：`退款编号 + SKU + 商品 + 规格`，同一售后单重复 SKU 会先合并。
- 平台商品：`商品 ID + SKU ID`。
- ERP 货品：`系统商品 ID + 系统 SKU ID`。
- 库存：`系统 SKU ID`。
- 店铺利润：`日期 + 平台 + 店铺`。

所有写入均为 upsert：已存在就更新，不存在才新增。日常同步会回看历史窗口，既能接收订单状态变化，也能接收退款变化，同时不会重复创建记录。

日同步会写入系统同步日志和日利润覆盖记录；存在单条写入失败、月份覆盖不完整或利润对账不一致时，工资结算批次会自动阻断。人员配置表的 `参与工资结算` 必须明确填写“是”，系统默认不把人员纳入发薪；按当前需求应只启用两个店铺、每店一人。

## 推荐自动任务

- 每天 02:00：执行 `npm run sync`，同步全部业务表。
- 每月 15 日 03:00：再次执行 `npm run sync`，确认上月利润完整后自动月结。
- 每天任务失败时：保留 `logs` 中的失败原因并通知负责人，不继续用不完整利润人工发工资。

Windows 可在“任务计划程序”中建立两个任务；Linux 可用 cron。生产环境建议把 `.env` 放在仅运行账号可读取的目录，并为飞书应用只授予多维表格所需权限。

## 上线检查

1. `npm test` 全部通过。
2. `npm run check` 同时返回 `kdzs: true` 和 `feishu: true`。
3. 先用较近的 `SYNC_START_DATE` 做小范围全量测试，核对 10 笔订单、退款和店铺利润。
4. 对账通过后再扩大日期范围。
5. 月结前核对 ERP 店铺利润合计与飞书 `02_店铺利润明细` 合计一致；不一致时不得结算。

客户可见且唯一生效的业务口径在飞书 `15_工资规则说明`；[docs/客户规则说明.md](docs/客户规则说明.md) 只是同内容的部署镜像。
